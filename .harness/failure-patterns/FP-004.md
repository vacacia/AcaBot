# FP-004 给 LLM 的工具协议写成黑盒

## 现象
scheduler 一开始把 `schedule.spec` 暴露成了一个很宽泛的 object，没有把不同 kind 的字段形状说清楚。
真实模型在干净上下文里仍然连续构造出：
- `one_shot.spec.delay = "120s"`
- `one_shot.spec.at = ...`
- `one_shot.spec.fire_at = "2025-..."`
而后端当时只接受 `fire_at: number`。

## 为什么错
对 LLM 来说，黑盒 schema 等于“自己猜协议”。
如果 tool desc / schema 不明确，模型就会按自然习惯发 `delay`、`at`、日期字符串，而不是你心里想的唯一格式。

## 代价
- 真实 LLM 验收反复失败
- 容易误判成模型能力问题
- 其实是 tool contract 设计问题

## 以后怎么避免
1. tool schema 不要只写一个裸 `object`，要把分支结构展开。
2. 对高概率变体同时做两件事：
   - desc/schema 明确标准写法
   - 后端兼容常见别名和自然格式
3. 验收时一旦看到模型稳定地“猜错同一类参数”，先反查 tool contract，而不是先怪模型。

## 一句话规则
**给 LLM 的工具协议不能写成黑盒；要么写清楚，要么做好兼容，最好两者都做。**
