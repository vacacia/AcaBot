# FP-001 把 fake agent 当成真实 LLM 验收

## 现象
为了补 scheduler 的 E2E，我先写了一组 runtime 级链路测试：
- event 进入 RuntimeApp
- fake agent 直接调用 `tool_executor("scheduler", ...)`
- scheduler 触发 synthetic event
- fake agent 再次处理

这组测试验证了 runtime/tool/scheduler 主链路，但我一度差点把它当成“完整验收”。

## 为什么错
用户要的不是“工具能被调到”，而是“LLM 在当前 system prompt 和 tool desc 约束下，真的会选择并正确使用这个工具”。

fake agent 跳过了这些关键设计面：
- tool description 是否足够清楚
- system prompt 是否正确引导
- LLM 是否会选对工具
- LLM 是否会构造对的参数
- LLM 在被 scheduler 唤醒后是否会给出合理回复

所以 fake agent 只能算 runtime 链路测试，不能算真实 LLM E2E 验收。

## 代价
- 容易过早汇报“E2E 已补”
- 容易漏掉 prompt / desc 设计问题
- 会让验收结论失真

## 以后怎么避免
1. 只要 AGENTS.md 里写了“必须要 LLM 响应”，就不能用 fake agent 充当最终 E2E 结论。
2. 汇报时必须明确区分：
   - runtime 链路 E2E
   - 真实 LLM E2E
3. 如果当前只是 fake agent 测试，措辞只能说“链路测试已补”，不能说“完整验收完成”。

## 一句话规则
**fake agent 只能证明 runtime 链路，不能证明 prompt / desc / LLM 设计有效。**
