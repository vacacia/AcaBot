# FP-005 SQLite 写失败会把后续调度一起毒死

## 现象
一次定时任务触发 synthetic event 时，runtime 在创建 run 记录阶段报：
- `sqlite3.OperationalError: database is locked`

表面上看像是“18:30 的 one-shot 没执行”，但真实情况是：
- 前一次写入失败把 SQLite 事务挂在连接上
- 后续别的 store 连接继续写时持续撞锁
- scheduler、run manager、event store 会一起表现异常

## 为什么错
runtime 里不同 store 各自持有 SQLite 连接。
如果还用默认 deferred 事务模式，某次写入抛出 `IntegrityError` / `OperationalError` 后，事务可能不会自动收尾；
连接继续持有写锁，其他连接随后就会看到 `database is locked`。

另外，scheduler 表结构一旦和代码期望不一致（例如历史库里 `scheduled_tasks.next_fire_at` 还是 `NOT NULL`），恢复阶段的 `disable(next_fire_at=None)` 也会再次制造失败写入，把问题放大。

## 代价
- one-shot / interval / cron 任务看起来“创建成功但不触发”
- runtime 重启恢复后仍可能继续异常
- 真实 LLM E2E 会被误判成 scheduler 逻辑问题，实际上是持久化层被毒死

## 以后怎么避免
1. SQLite store 默认使用 autocommit，并显式设置 `busy_timeout`。
2. 任何共享数据库的 store 都要把“失败写入后其它连接还能继续写”写成测试。
3. schema 发生语义变化时，不只加列；还要检查旧列约束是否与新语义冲突（如 `NOT NULL` -> 可空）。
4. 线上出现“调度没执行”时，先查日志里是否已有更早的 SQLite 写失败，不要只盯着 scheduler 表面状态。

## 一句话规则
**SQLite 一次失败写入如果不收尾，会把后续调度链路一起毒死；先防锁污染，再谈 scheduler。**
