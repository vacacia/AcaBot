# 长期记忆怎么和现有框架交互

这一篇只讲一件事：

- 类似 SimpleMem 这种“从消息记录里自己提取内容、自己判断保留什么、在检索时再把记忆提供给前台 runtime”的长期记忆组件，应该怎样接进 AcaBot 当前框架

这一篇不展开：

- 长期记忆内部怎么抽取
- 长期记忆内部怎么合并
- embedding / rerank / 向量库
- prompt 和算法细节

---

## 先讲结论

AcaBot 里，长期记忆更适合收成下面这个形状：

- 长期记忆是一个 **独立的长期记忆组件**
- 它自己读取 `event / message facts` 或消息记录
- 它自己决定保留什么、写到哪里、什么时候更新
- 前台 runtime 在需要上下文时，只通过统一 retrieval 入口向它取记忆

换句话说：

> 长期记忆和前台 runtime 的接缝，重点在“检索时怎么接进主线”，不在“前台这一轮怎么决定要不要写长期记忆”。

这也意味着：

- `SessionRuntime` 不负责决定长期记忆要不要提取
- `MemoryBroker` 不负责决定长期记忆要不要写、写到哪一层
- 长期记忆组件自己消费消息记录，自己完成提取和写入

---

## 这篇在讨论什么

这里说的“长期记忆”，专指这一类能力：

- 它面对的是消息事实和消息记录
- 它会在后台持续地看这些记录
- 它自己分析哪些内容值得长期保留
- 它把保留下来的结果存成自己可检索的长期材料
- 前台 runtime 在某一轮需要时，再向它取回相关内容

它和下面几层不是同一个东西：

- `thread working memory`
  - 当前 thread 的短期上下文
- `event / message facts`
  - 平台真实发生过什么、系统真实发送过什么
- sticky note
  - 可直接维护、可直接编辑的稳定笔记
- `/self`
  - Aca 自己的连续性文件区

长期记忆在这里更像：

- 一个建立在事实层之上的独立记忆系统
- 一个平时自己整理、用时再被检索的长期材料库

---

## 长期记忆和现有框架的真实接缝

如果只看框架交互，长期记忆和 AcaBot 的关系可以直接拆成两条线：

- 写入线
- 检索线

其中真正要收硬的，是这两条线的边界。

## 一、写入线：长期记忆组件自己消费消息记录

长期记忆的写入，不应该挂在前台 run 的收尾决策上。

更顺的形状是：

`event / message facts -> 长期记忆组件 -> 长期记忆自己的存储`

也就是说：

- 前台 runtime 负责把事实记录保存下来
- 长期记忆组件自己去读这些记录
- 长期记忆组件自己做分析、筛选、合并和写入

这样长期记忆的写入主权属于长期记忆组件自己，而不是前台 runtime。

### 1. `SessionRuntime` 不负责长期记忆组件内部决策

`SessionRuntime` 负责的是：

- 当前消息怎么路由
- 当前消息怎么进入前台 run
- 当前消息是否回复、是否只记录

它不应该负责：

- 这条消息要不要进入长期记忆
- 这条消息应该写到哪个长期记忆 scope
- 这条消息应该带哪些长期记忆标签

这些判断本质上属于长期记忆组件自己。

因为对长期记忆来说，更合理的输入不是“当前 run 的一份即时判断”，而是：

- 一段完整消息记录
- 更长时间范围内的上下文
- 长期记忆组件自己维护的状态

### 2. `ThreadPipeline` 不负责触发长期记忆写回

前台 pipeline 的职责是：

- 跑完一次前台响应主线
- 组织本轮上下文
- 调模型
- 发送动作

它不应该在 run 收尾时再去决定：

- 要不要写长期记忆
- 写入哪一类长期记忆
- 触发哪一种长期记忆分析器

长期记忆应该是一个和前台 run 解耦的消费者。

### 3. 长期记忆组件自己决定写什么

长期记忆组件应该自己拥有下面这些判断：

- 要不要保留一段内容
- 这段内容是废话、事实、关系材料还是群体背景
- 应该合并进已有记忆，还是形成新记忆
- 应该什么时候写入

这类能力都依赖长期记忆自己的处理逻辑。

所以这里最重要的原则是：

> 前台 runtime 只负责把事实留下来；长期记忆组件自己决定怎样把事实变成长期记忆。

---

## 二、检索线：前台 runtime 在需要时统一取回长期记忆

长期记忆和前台 runtime 的正式主接缝，应该放在 retrieval 这一侧。

这一条线更适合收成：

`RetrievalPlanner -> MemoryBroker.retrieve -> 长期记忆 retriever -> ContextAssembler`

这里前台 runtime 真正关心的是：

- 当前这轮要给模型补什么上下文
- 当前有哪些长期记忆来源可以问
- 这些来源返回了哪些内容块
- 这些内容块最后怎样排进模型输入

### 1. `RetrievalPlanner` 只准备这一轮检索现场

`RetrievalPlanner` 更适合负责：

- 当前 run 的 query text
- 当前有效 working summary
- retained history
- actor / channel / thread 这些检索现场信息

它表达的是：

- 这一轮前台在问什么
- 这一轮前台现在处在什么上下文里

它不负责：

- 替长期记忆做写入判断
- 替长期记忆做保留判断
- 决定长期记忆内部怎样命中

### 2. `MemoryBroker` 只负责统一 retrieval

`MemoryBroker` 在这里的正式职责应该很窄：

- 接收当前 run 的 retrieval request
- 统一去问各个可用的记忆来源
- 收集它们返回的 `MemoryBlock`
- 交给后面的上下文组装层

它不负责：

- 判断长期记忆要不要写
- 判断写进哪一种长期记忆
- 主动安排长期记忆组件怎样消费消息记录

也就是说，`MemoryBroker` 是 **retrieval broker**，不是长期记忆写入调度器。

### 3. 长期记忆组件以 retriever 的身份接进来

长期记忆组件接前台主线时，最顺的身份是：

- 一个长期记忆 retriever

它收到 retrieval request 之后，自己决定：

- 这一轮要不要返回内容
- 返回哪些记忆
- 每块内容怎样变成 `MemoryBlock`

前台 runtime 不需要知道它内部怎么命中，只需要知道：

- 它是一个可用的长期记忆来源
- 它能在 retrieval 时返回统一格式的内容块

### 4. `ContextAssembler` 只消费 retrieval 结果

长期记忆进入 prompt 的入口，应该统一收在 `ContextAssembler`。

对应分工是：

- 长期记忆 retriever 返回 `MemoryBlock`
- `ContextAssembler` 只负责把这些块排进最终上下文

这样 assembler 关心的是：

- 这块内容放到哪个 slot
- 这块内容在整体上下文里排在哪里

它不需要理解长期记忆组件内部怎样工作。

## 长期记忆组件最自然的物理位置

如果从系统结构上看，长期记忆组件更像前台 runtime 旁边的一条独立侧线，而不是前台 pipeline 里的一个步骤。

更自然的形状是：

```text
Gateway / Outbox
  -> event facts / message facts
       -> 长期记忆组件自己读取记录
            -> 长期记忆自己的存储

前台 run
  -> RetrievalPlanner
  -> MemoryBroker.retrieve
  -> 长期记忆 retriever
  -> ContextAssembler
  -> ModelAgentRuntime
```

这样一来：

- 写入线是 ***异步的*** 、独立的
- 检索线是按需的、***同步*** 进主线的

这比把长期记忆整体塞进一次前台 run 里更稳。

---

## builtin 还是 plugin，重点不在包装方式

这里最重要的不是“文件放 builtin 目录还是 plugin 目录”，而是责任线对不对。

只要边界正确，长期记忆组件既可以是：

- runtime 自带组件
- 也可以是后面接进来的扩展组件

真正要收硬的是下面这件事：

- 它必须自己消费消息记录
- 它必须自己决定保留和写入
- 它在前台主线里只以 retriever 的身份提供记忆

所以包装方式是第二层问题，边界才是第一层问题。

如果以后把它做成 plugin，更顺的方式也应该是：

- plugin 启动一个长期记忆组件
- 这个组件自己读消息记录
- 这个组件向前台注册一个 retriever

而不是：

- plugin 直接改 `SessionRuntime`
- plugin 直接改前台 run 的写入决策
- plugin 在 pipeline 收尾时替长期记忆下判断

## 和其他记忆层的关系

为了让长期记忆边界保持稳定，它和其他几层的关系可以直接收成下面这样。

### `/self`

`/self` 是 Aca 的自我连续性区。

它可以和长期记忆一起通过 retrieval 进入前台上下文，但 `/self` 的维护方式仍然属于 `/self` 自己。

### sticky note

sticky note 是稳定、可直接维护的笔记。

它和长期记忆都可以通过统一 retrieval 入口进入上下文，但 sticky note 仍然是 sticky note 服务自己维护，不并入长期记忆组件。

### thread working memory

thread working memory 继续只负责当前 thread 的短期上下文。

它可以作为长期记忆检索时的现场输入，但它本身不变成长记忆。

### event / message facts

事实层继续负责保存真实发生过什么。

长期记忆组件从这里取原料，但不替代事实层。

## 对控制面和 WebUI 的含义

如果按这套边界继续往下长，控制面和 WebUI 更适合围绕下面这些点扩展。

### 1. 展示长期记忆来源

控制面更适合回答：

- 当前有哪些长期记忆来源
- 哪些来源当前可检索
- 每个来源最后返回了哪些内容块

### 2. 展示长期记忆组件自己的产出

控制面更适合展示：

- 长期记忆组件当前保存了哪些长期材料
- 这些材料属于哪个 actor / channel / relationship 方向
- 最近一次更新发生在什么时候

### 3. 不把“是否写入长期记忆”做成 session 配置

如果长期记忆组件自己消费消息记录，那控制面的重点就不应该是：

- 某个 session 要不要写长期记忆
- 某个 surface 默认写哪个长期记忆 scope

更适合的控制面方向是：

- 长期记忆组件是否启用
- 它看哪些消息记录范围
- 它自己的检索和展示策略

---

## 最后把最终形态收成一句话

在 AcaBot 里，长期记忆最顺的交互方式是：

> **前台 runtime 只负责保存事实，并在 retrieval 时统一向长期记忆组件取内容；长期记忆组件自己消费消息记录、自己决定保留和写入、在需要时再以 retriever 的身份把记忆交回主线。**

这样责任线是清楚的：

- 前台 runtime 不替长期记忆下写入判断
- `MemoryBroker` 不替长期记忆做写入调度
- 长期记忆组件自己拥有长期记忆的形成过程
- 前台主线只在需要上下文时统一消费结果
