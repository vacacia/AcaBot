# AcaBot 项目理解与核心共识

这篇文档不是实现细节，也不是功能清单。它记录的是当前已经达成、而且应该在后续对话里持续成立的项目级共识。

当前第一版共识聚焦在 `self`、不同运行链路的边界，以及长期信息应该怎么分层。之所以先把这部分单独写出来，是因为它已经不是“某个小功能的讨论结果”，而是会直接影响 WebUI、prompt 组装、memory 设计、后台接入方式和后续产品理解的核心前提。

以后无论谁新开一个对话，只要先读这一篇，就能更快理解 AcaBot 里 `self`、memory、对外交互链路和维护链路到底该怎么分。

## 项目级结论

`self` 是 AcaBot 里 Aca 的稳定自我设定区。

它描述的是：

- Aca 是谁
- Aca 的长期人格特征和说话风格是什么
- Aca 的角色边界和长期职责是什么
- 这些东西里，哪些和具体群、具体用户无关

所以 `self` 的定位不是“杂项长期记忆”，也不是“所有能长期保存的东西”。它更接近系统里 Aca 的稳定自我底座。

当前共识是：

- `self` 只进入对外交互链路
- `self` 由系统对外运行这一侧维护
- `self` 固定注入对外交互上下文
- `task` 不再单独作为一层考虑，而是并入 `self`
- 后台可以看见 `self` 文件，但不把它注入后台
- `sticky_note` 是当前真正暴露给产品和 WebUI 的 memory 形态
- `sticky_note` 只使用 `user` 和 `channel` 两种 scope
- relationship memory / channel memory 仍是概念层级，当前不做产品化暴露

## Self

### 概念

`self` 是系统里 Aca 对“我是谁”的稳定表达层。

它对应的是那些不应该随着群聊切换、用户切换而改变的东西，例如：

- identity：Aca 的身份、角色、自称、核心定位
- soul：Aca 的长期气质、价值倾向、表达风格、行为边界
- state：Aca 当前希望长期维持的自我状态、持续中的承诺、公开侧可见的长期约束
- task：Aca 当前正在做什么、接下来要做什么、有哪些需要跨上下文持续携带的实时状态

这些内容的共同特点是：

- 它们属于 Aca 本身
- 它们和“当前正在跟谁说话”无关
- 它们应该稳定存在，而不是每个线程临时拼装
- 它们应该能被人类直接查看和修改

这里要特别注意：`self` 里既有长期稳定的内容，也可以有实时状态内容。当前共识里，`task` 不再单独作为一层，而是作为 `self` 的一个文件存在，用来记录 bot 当前在做什么、接下来要做什么，以及一些需要全局持续了解的状态。

所以 `self` 更像“自我真源”，而不是某种隐藏在实现内部的 prompt 片段。

### 不是什么

`self` 不是下面这些东西：

- 不是某个用户的关系记忆
- 不是某个群的共同背景
- 不是某个线程的 working memory
- 不是平台事件事实
- 不是维护链路自己的工程提示词
- 不是 sticky note 的替代品

如果把这些东西都混进 `self`，会马上失去边界：

- 人格层会和关系层混在一起
- 稳定自我会和临时对话上下文混在一起
- 不同运行链路需要的材料会混在一起

这正是之后最容易把系统做乱的地方。

### 在项目里解决什么问题

`self` 解决的是 Aca 在对外交互时的稳定性问题，而不是维护链路的执行问题。

对外运行这一侧需要的是：

- 对外聊天
- 维持可识别、连续的自我
- 在不同群和私聊里保持稳定的自我表达
- 在这个稳定自我之上，再叠加人际关系和群体差异

没有这层，Aca 的稳定自我只能靠 prompt 碎片、线程上下文和临时记忆拼出来，长期会很不稳。

### 文件组织

当前共识不是“只放几个散文件”，也不是“完全自由文件系统”。

更稳的结构是：

- 使用固定目录 `/self/`
- 在里面保留固定主文件
- 同时允许有附加文件

主文件代表系统里 Aca 的稳定自我底座，例如：

- `identity.md`
- `soul.md`
- `state.yaml`
- `task.md`

这些主文件应被视为 `self` 的核心真源。

附加文件可以存在，但它们是围绕 `self` 的补充材料，不应反过来破坏主文件作为稳定底座的地位。

其中：

- `identity / soul / state` 更偏长期稳定内容
- `task` 更偏实时状态内容

但它们都属于 `self`，都会进入对外交互上下文。

这里再补一条很重要的实现共识：

- `MemoryBroker` 只负责检索规则
- 每类 memory 的来源由自己管理

所以 `self` 不必被理解成“必须归 `MemoryBroker` 管”。只要它的规则等价于“每个对话、每次 run 都会进入上下文”，不管底层是不是经过 `MemoryBroker`，都符合当前共识。

## 运行链路边界

### 为什么后台不注入 `self`

维护链路的职责不是扮演 Aca 对外交流时的自我，而是保证系统运行和演化。

在当前构想里，后台是独立维护的工程执行面。它的核心任务是：

- 看仓库
- 改代码
- 改配置
- 跑测试
- 修文档
- 处理运行时维护问题

所以后台需要的是清晰的工程目标、权限边界和维护规则，而不是 `self` 里的自我设定。

把 `self` 注入后台没有实际帮助，反而会干扰后台工作。原因很简单：

- `self` 里的自我特征并不能帮助后台做工程判断
- `self` 里的表达风格并不能帮助后台维护系统
- 这些自我设定对后台执行维护链路没有正收益
- 这些内容会额外占用上下文预算，干扰后台聚焦维护任务

换句话说，后台不是“带着同一套 `self` 的另一个视角”，而是项目里的独立维护执行面。

因此当前共识是：

- 后台对 `self` 有文件级可见性
- 但后台不注入 `self`
- 后台如果需要读取 `self`，那是把它当作项目材料查看，而不是把它当作自己的工作设定

### 可见性和注入是两回事

后台可以看见 `self`，不等于后台应该携带 `self` 工作。

这里要分清两件事：

- 文件可见性
- 运行时注入

后台因为位于仓库根目录和运行时目录之上，所以天然拥有文件可见性；但运行时是否把某类材料注入到后台上下文里，是另一层决策。当前共识是：后台保留可见性，但不做 `self` 注入。

## 长期信息分层

### 分层总览

当前已经达成的分层共识如下：

- `/self`
  - Aca 的稳定自我设定
  - 和具体人、具体群无关
  - 固定进入对外交互链路
- sticky note
  - 当前真正暴露给产品和 WebUI 的 memory 形态
  - 目前只依附于 `user` 和 `channel` 两种 scope
  - 在需要时按对应 scope 进入上下文
- relationship memory
  - 表示和具体对象有关的长期记忆方向
  - 当前代码和产品层都还没真正做出来
- channel memory
  - 表示和具体群体有关的长期记忆方向
  - 当前代码和产品层都还没真正做出来
- thread working memory
  - 最近几轮对话的短期上下文
- event / message facts
  - 平台真实发生过什么、系统真实发送过什么

这里最关键的几条是：

- `self` 的规则等价于每轮永远命中、每轮进入上下文
- sticky note 在需要时按对应 scope 进入上下文
- relationship memory / channel memory 目前还是概念层，不先做成产品对象

这些东西不能混成一个层。

### `self`

`self` 是稳定自我设定真源，和具体人、具体群无关，固定进入对外交互链路。

### sticky note

sticky note 是当前真正进入产品层和 WebUI 的记忆形态。

它的定位是零碎但长期有用的笔记。例如：

- 这个群的群主是谁
- 这个群的风格是什么
- 这个群主要在聊什么
- 这个群有哪些黑话

bot 可以写入 sticky note，让自己以后还能理解这些东西。

当前共识里，sticky note 对每个 scope 都分成两块：

- 只读区
  - 人工编写
  - 百分百真源
  - bot 不允许更改
- 可编辑区
  - 允许 bot 写入和更新

注入规则也已经定死：

- `readonly` 注入
- `editable` 也注入

从产品和 UI 的角度看，这最接近“两个文件 / 两个分区”的组织形式。

### relationship memory

relationship memory 目前仍然是概念层级，不是第一版产品对象。

虽然名字已经存在，但当前代码和体验都还没有把它真正做成一套可用的独立记忆层，所以 WebUI 暂时不展示它。

### channel memory

channel memory 目前仍然是概念层级，不是第一版产品对象。

它表达的是和具体群体有关的长期记忆方向，但当前代码和体验都还没有把它真正做成一套可用的独立记忆层，所以 WebUI 暂时不展示它。

### thread working memory

thread working memory 是最近几轮对话的短期上下文，不承担稳定自我和长期关系的职责。

### event / message facts

event / message facts 负责记录真实发生过什么和真实发送过什么，本质上就是消息聊天记录。

后续可以在 WebUI 里增加专门的查询页，方便查看聊天记录，但当前不把它作为记忆编辑页面的一部分。

## WebUI 与实现约束

### WebUI

WebUI 必须直接把这层暴露出来，而不是把它藏进抽象配置项里。

当前共识是：

- WebUI 要使用 Vue 重做
- WebUI 中要有明确可见的 `self` 编辑区域
- 用户应能直接看到 `/self` 主文件内容
- 用户应能直接修改这些文件
- WebUI 中要有单独的 `Memory` 页面来展示 sticky notes
- `self` 不应只以“一个大 prompt 文本框”的形式出现

原因很简单：

- `self` 是系统真源的一部分，不是实现细节
- 既然它是长期真源，就必须具备直接查看和直接修改能力
- 如果 UI 只暴露抽象字段，用户很难真正掌控这层真源

### 对实现的直接约束

虽然这篇文档不是实现方案，但它已经对后续实现给出几条硬约束：

- 不要把 `self` 设计成维护链路材料
- 不要把 sticky note 和 `self` 混成一个东西
- 不要把尚未落地的 relationship / channel memory 先硬做成产品页面
- 不要把 `self` 退化成一个不可见的 prompt 拼装细节
- 不要让 WebUI 只能改抽象配置而看不到文件真源
- 不要把后台“能看见 `self`”误解成“后台应该注入 `self`”

## 一句话版本

`self` 是 AcaBot 里的核心真源，既包含长期稳定的自我设定，也包含 `task` 这类实时状态；sticky note 是当前真正暴露给产品层的 memory 形态；后台可以看见这些材料，但不注入 `self`，因为维护链路不应被这层材料干扰。
