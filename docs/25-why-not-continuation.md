# 为什么当前主线不做 continuation

这篇文档只回答一件事：

**为什么现在这条 runtime 主线不能把一次 `run` 的私有执行过程，直接延续成后续消息里的 generic continuation。**

这里说的 continuation，不是 approval 这种已经被正式定义过的中断恢复；而是更宽泛的想法：

- 这次 `run` 里工具做到一半
- bot 先给用户发一条中间状态
- 用户下一条消息回来后
- 系统继续吃上一次 `run` 的隐藏工具上下文，接着往下跑

当前主线不承载这种 generic continuation。


## 为什么 generic continuation 立不住

### 1. 当前 `run` 私有态没有稳定 replay 契约

一次 `run` 里的这些东西，今天都属于执行现场：

- 组装出来的模型消息
- tool loop 中间态
- 工具原始返回
- 临时变量
- 当前轮的局部判断

这些内容没有被定义成“下一条用户消息还能继续复用的正式对象”。

如果直接把它们当 continuation 接下去，系统就会失去边界：

- `run` 不再只是一次执行
- `thread` 不再只是公共消息
- 工具过程开始变成半公开、半私有、可续跑的混合状态

这会让恢复、取消、审计和并发全部变脏。

### 2. 工具缺字段，不等于系统应该挂起等用户

工具校验只能说明：
- 缺了哪些字段
- 参数是否合法
- 当前调用能不能执行

它说明不了：
- 模型是不是漏填了本来能从上下文里补出来的参数
- 当前是不是该追问用户
- 当前是不是该换个方案而不是继续调这个工具

所以“工具缺字段”本身，不能自动变成 continuation。

continuation 不是工具层自动推出来的，而是 agent 做出显式决策后，runtime 再把这个决策物化成正式状态。

### 3. `job` 不能靠 runtime 猜

“这件事是不是该变成一个可继续、可取消、可查进度的 job”，当前主线也不能靠观察普通 run 去猜。

### 4. 群聊和私聊不是同一种前台模型

generic continuation 最大的问题之一，是它默认把所有前台消息都当成“可以吃进同一个隐藏执行现场”的输入。

这对群聊不成立。

群聊天然是：

- 多 actor
- 多消息并发
- 乱序插话
- bot 的状态消息不一定真的被下一条消息承接

所以群聊更稳的边界是：

- `event -> short run`
- 新消息就是新 run
- 共享的只有公共 `thread` 历史和少数显式 continuation

私聊则不同。

私聊如果要支持“边做工具调用边发状态、用户中途回一句还能接上”，更合理的方向不是把 generic continuation 塞回 `run`，而是直接承认：

- 私聊前台需要一个稳定的 foreground worker
- 用户消息进入这个 worker 的 inbox
- worker 自己维护等待输入、当前进度和可继续的前台状态

也就是说：

- 群聊适合 short run
- 私聊如果要连续前台体验，适合 worker

foreground worker 的完整设计在 `docs/26-foreground-worker.md`。
