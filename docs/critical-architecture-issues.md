# 当前阶段最严重的架构问题

这一篇不是完整问题清单，也不是未来蓝图。

它只做一件事：在重新审视现有框架之后，明确指出**现在最该优先处理的结构性问题是什么**，以及为什么它比别的问题更严重。

如果你只打算记住一句话，那就是：

> 当前最严重的问题，不是某个单点 bug，也不是某个页面不完整，而是**能力边界和抽象边界还没有完全收口**。

这会直接影响系统的可信度、后续演进成本，以及 WebUI / profile / runtime 之间的语义一致性。

## 先讲结论

当前最严重的三个问题，可以这样排:

1. **第一严重问题：能力边界和抽象边界没有完全收口**
2. **第二严重问题：`runtime` 继续中心化，子域边界更多靠文档维持**
3. **第三严重问题：产品概念、配置概念、底层实现概念还没有完全对齐**

这三条里，第一条是根因级问题。

后两条虽然也严重，但更多像是第一条长期没有收口之后，系统自然长出来的症状和放大器。

## 1. 第一严重问题：能力边界和抽象边界没有完全收口

这里说的不是“抽象很多”，而是更具体的一件事：

系统已经在文档、WebUI 和 profile 配置层表达出一套很明确的能力组织方式，例如：

- tool 是给模型按需调用的能力接口
- plugin 是系统级扩展点
- skill 是能力包 / SOP
- subagent 是可委派的独立 worker
- profile / assignment 决定谁能看到哪些能力

但这套边界在真实运行时里，还没有完全变成唯一权威的执行规则。

### 现象

目前至少有几类现象已经说明这个问题是真实存在的。

#### 1. direct subagent delegation 还能绕开部分能力边界

`delegate_subagent` 当前支持直接传 `delegate_agent_id`。

这意味着某些路径会优先看 executor registry，而不是先严格经过当前 profile 的 assignment / delegation policy。

结果就是：

- WebUI 上看起来没开放给当前 agent 的 subagent
- profile 上看起来没分配给它的能力
- 在某些 direct delegation 路径里，运行时仍然可能调到

这不是普通实现瑕疵，而是“配置表达的规则”和“执行时真正生效的规则”之间出现了缝。

#### 2. `skill` 主线已经收口，但能力模型还没有完全统一

当前 skill 这条线已经比原来清楚很多了：

- runtime 会递归扫描 skill 目录
- prompt 会先放 `<system-reminder>` skill 摘要
- 模型调用的是 `Skill(skill=...)`
- runtime 会返回 `Launching skill: ...` 和 `Base directory for this skill: /skills/...`
- 后续资料通过 `/skills/...` 继续读

所以这块现在已经有一条稳定主线。

现在真正还没完全收干净的问题，不是 skill 自己不会工作，而是：

- skill
- tool
- plugin
- subagent

这几类能力的总模型还分散在多份文档和多层运行时规则里。

也就是说，skill 这条主线本身已经清楚了，但系统整体的“能力边界总图”还没有完全压成单一规则面。

#### 3. `tool / plugin / skill / subagent` 这组概念还没有完全收成单一能力模型

文档层已经对四者做了区分，这是对的。

但从系统实际演进趋势看，它们都在回答同一个问题：

> bot 到底能借助什么能力做事，这些能力又是怎样暴露、隔离、委派和组合的？

问题在于，这个总问题现在还没有一个完全收口的“单一权威模型”。

所以后来者经常会遇到这种情况：

- 这个功能像 tool，但也可以做成 plugin
- 这个能力像 skill，但又需要脚本和资源
- 这个任务像 subagent，但又只是一次能力扩展

如果这些判断长期依赖经验和文档提醒，而不是依赖一套明确、硬性的运行时规则，那系统就会慢慢进入“概念越来越多，但边界越来越靠口头维护”的状态。

### 为什么这是当前最严重的问题

因为它伤的不是“某个功能”，而是系统的**可信边界**。

一个平台型系统最怕的不是能力不够，而是：

- 配置层说一套
- WebUI 暴露一套
- 文档解释一套
- 运行时真正执行时又是另一套

一旦出现这种情况，后果会比单点 bug 更麻烦：

#### 1. 能力隔离会失真

用户或维护者以为 profile 已经限制了某个能力，但运行时实际上还有更宽的直达路径。

#### 2. 后续抽象会建立在不稳定地基上

如果旧边界还没收紧，就继续往上长更多 skill、subagent、backend bridge 或 workspace 能力，后面只会更难收。

#### 3. AI 和维护者会越来越依赖“记住例外”

系统越复杂，越不能靠“这个地方其实还有个特殊路径”来维持理解。

一旦后续要继续让 AI 直接改代码，这种不一致会非常容易让改动走偏。

### 根因判断

这个问题不是某个开发失误导致的，而是系统阶段变化带来的。

更准确地说，它来自三件事叠在一起：

#### 1. 系统已经从“单一 bot 工程”长成“平台型 runtime”

一开始很多能力可以先接上再说，但现在已经不是那个阶段了。

现在系统里已经同时有：

- capability visibility
- assignment
- delegation
- tool broker
- plugin manager
- skill catalog
- subagent executor
- backend bridge

当能力种类和入口变多之后，原来那些“先这样接一下也行”的做法，就会开始变成结构性负担。

#### 2. 一部分概念是产品层语言，一部分概念是运行时接线语言

比如：

- WebUI / profile 关心“这个 agent 能看到什么能力”
- runtime 关心“这次执行具体怎么 resolve 到 handler / executor”

如果中间没有一个非常硬的统一解析面，系统就容易变成：

- 上层说的是能力边界
- 下层做的是接线便利

最后两边语义不完全重合。

#### 3. 现阶段更像“抽象已经命名完成，但执行语义还没完全封口”

也就是说，现在的问题不是没抽象，而是抽象已经有了，但它们还没有完全变成唯一生效的硬规则。

这比“还没抽象”更危险，因为它会制造一种“系统已经很规整了”的错觉。

### 建议整改方向

短期最值得做的，不是大重构，而是先把“能力边界”收成一个真正可信的规则面。

建议顺序：

1. **先收紧 direct delegation 路径**
   - 任何 subagent 调用都应该先经过统一的可见性 / assignment / delegation 校验
   - 不要再允许“registry 里有就能直接调”的旁路

2. **保持 `skill` 继续沿当前主线演进**
   - 维持它作为“可读能力包 / SOP”的位置
   - 继续补齐 references / scripts / 渐进加载这些围绕资料包本身的能力
   - 不要再把它重新扩成别的运行时能力模型

3. **把能力解析收敛到单一权威入口**
   - 谁能看到什么能力
   - 谁能调用什么能力
   - 谁能委派给谁
   - 最终都应该从同一套规则得出，而不是分散在 profile、plugin 注册、executor registry 和某些工具参数里各管一段

4. **为能力边界补测试，不只补文档**
   - 要有专门验证“不可见 subagent 不能被 direct delegation 调到”的测试
   - 要有验证 skill assignment / delegation mode 生效边界的测试

### 暂时不要怎么修

这里最忌讳三种修法：

#### 1. 不要一口气把整套能力系统重写成“万能 capability engine”

这很容易把当前已经能跑的东西全部掀翻，收益反而不稳定。

#### 2. 不要只补文档，不收执行路径

这类问题不是“解释不清”，而是“执行边界真的没封紧”。

#### 3. 不要继续增加新名词来绕开旧问题

如果旧的 `tool / plugin / skill / subagent` 边界还没收好，就不要再创造新的中间层名词去承载同类能力。

## 2. 第二严重问题：`runtime` 继续中心化，子域边界更多靠文档维持

这条问题不是说 `runtime` 不该是核心。

问题在于：现在真正重要的事情，几乎都还在往 `runtime/` 里汇聚。

它已经承担了：

- app / router / pipeline 主线
- memory
- tool broker
- plugin manager
- subagent
- control plane
- model registry
- reference backend
- computer
- backend maintainer 接线

目录上虽然已经拆了很多子域，但系统理解上，`runtime/` 仍然太像“真正有用的东西都在这里”。

### 为什么这会越来越疼

因为这会让系统逐渐依赖一种工作方式：

- 先读一圈 docs
- 再知道要从哪个入口切
- 再小心不要跨错边界

这说明文档很强，但也说明代码本身还不够自解释。

现在这套架构还能维持，靠的是：

- 文档质量高
- 维护者对边界很敏感
- 目录拆分已经做了一半

但如果后续继续加能力，而 `runtime` 的真实职责不再继续收缩，后面会更像一个“大中台”，而不是一组边界稳定、可独立演进的子域。

### 特别值得注意的两个方向

#### 1. `computer` 已经在向第二条基础设施主线发展

它不只是 file tool，也不只是 shell session，而是同时管：

- workspace
- attachments staging
- exec / session
- backend 抽象
- skill mirror
- tool adapter 接缝

这种子系统如果继续长下去，但仍然只是 `runtime` 里的一个“普通子目录”，后面会越来越像另一个中台。

#### 2. backend maintainer 方向已经不是小功能了

当前 backend 集成已经是真实接线，而不是概念草图。

这条线如果继续发展成：

- canonical session binding
- query / change 分流
- operation / artifact 投影
- 后台模式状态机

那它本质上已经是一个正式子域，不该长期以“在 runtime 里接几层”这种方式继续长。

### 建议整改方向

- 继续保持 `app.py` 和 `pipeline.py` 的最小主线职责，不要往里塞更多特例
- 对 `computer`、backend、control plane 这类已经明显长大的子域，继续强化“本域入口”和“本域协议”，减少外部直接跨文件借道
- 文档继续保留，但不要满足于“文档解释得清楚”，要让代码边界也越来越能自解释

### 暂时不要怎么修

- 不要为了“runtime 太大”就做一次大拆仓
- 不要只在目录上再拆一层文件夹，却不收真实职责
- 不要把“读文档顺序”当成长期替代边界治理的方法

## 3. 第三严重问题：产品概念、配置概念、底层实现概念还没有完全对齐

这一条不是安全性最强的问题，但它会不断拉高认知负担。

### 典型表现

#### 1. `self` 和 `soul` 的命名口径还没完全统一

代码已经开始往 `soul` 落，文档和项目共识里仍然大量保留 `self`。

这说明系统的概念演进已经发生了，但术语冻结还没完成。

#### 2. memory 的产品语言和底层模型之间还有一层翻译

产品上更自然的说法是：

- self / soul
- sticky note
- relationship memory
- channel memory
- thread working memory

底层当前更正式的模型是：

- `scope`
- `scope_key`
- `memory_type`

这不是错，但如果两层语言长期不完全对齐，AI、维护者和 WebUI 都要额外记住一层映射。

#### 3. WebUI 越来越依赖 shell / snapshot / 映射层

这说明后端真实结构和产品编辑模型之间还有距离。

这种映射层是必要的，但如果越来越多，也意味着：

- 系统内部模型并不是天然产品友好
- 控制面越来越依赖解释和转换，而不是直接围绕稳定资源模型工作

### 为什么它现在排第三

因为这条问题虽然会持续消耗理解成本，但它短期内还不一定直接破坏能力边界。

相比之下，第一条更接近“系统说自己有哪些规则，实际是不是按这套规则工作”的问题，所以优先级更高。

### 建议整改方向

- 尽快冻结一套术语表，明确哪些词是产品词，哪些词是底层词，哪些词已经废弃
- 给 memory 体系补一张稳定映射表，不要让后来者自己猜“产品分层”和 `scope + memory_type` 的关系
- WebUI 如果必须做 shell / snapshot 映射，也尽量让映射规则集中，而不是散在多个页面和控制面接口里

## 4. 为什么第一问题必须优先处理

如果只能优先修一类问题，应该先修第一条。

原因很简单：

- 第二条主要影响系统继续长下去时的可维护性
- 第三条主要影响认知成本和产品表达的一致性
- **第一条直接影响系统边界是否可信**

一个复杂系统可以暂时不够优雅，可以暂时有命名分叉，也可以暂时比较依赖文档。

但它不能长期处于这种状态：

- UI 上看起来能力被限制了
- 配置上看起来规则已经表达了
- 文档上看起来边界已经讲清楚了
- 运行时里却还有更宽的隐藏路径

这会让后续所有更高级的能力建设都建立在不稳定地基上。

说得更直白一点：

> 如果第一条不先收口，后面的系统演进就很容易变成“文档上越来越优雅，运行时上越来越靠补丁”。

## 5. 建议的短期整改顺序

建议按这个顺序处理：

1. **先修 subagent / delegation 的边界一致性**
2. **保持 skill 沿当前主线稳定演进**
3. **把 capability visibility / resolve / delegation 收敛到单一权威入口**
4. **为能力边界补回归测试**
5. **再继续处理 runtime 瘦身和术语统一**

## 6. 这篇文档和 `15-known-issues-and-design-gaps.md` 的关系

两篇文档的定位不同：

- `15-known-issues-and-design-gaps.md`
  - 更像问题清单
  - 记录当前已经确认的缺口
  - 方便后续改代码时顺手同步

- 这一篇
  - 更像优先级判断
  - 不是想列全，而是想明确“现在最该先怕什么”
  - 重点不是覆盖面，而是排序和判断

所以如果以后 `15` 继续增长，这一篇也不一定要跟着扩成总表。

它只需要持续回答一个问题：

> 以当前系统阶段看，什么问题最值得优先当成架构风险处理？

当前答案就是：

> **能力边界和抽象边界还没有完全收口。**
