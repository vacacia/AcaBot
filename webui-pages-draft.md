# AcaBot 后台控制台页面草图

这份文档写的是页面怎么长、怎么分区、用户在页面里做什么。

不是字段盘点表。

## A / B / C

- `A`：当前代码和接口已经基本支持，可以直接做页面
- `B`：底层能力已存在，但还缺一层控制面接口或产品化包装
- `C`：当前代码不支持，需要补后端契约

## 导航

### 主干直出

- 首页
- 模型供应商
- 模型预设
- 提示词
- Session
- 记忆
- 日志
- 系统

### 分组

- 扩展能力
  - 插件
  - 技能
  - 工具
  - SubAgent

### 约束

- 侧边栏默认展开
- 不做聊天工作区
- `平台` 不单独成页，相关配置并入 `系统`

## 首页

定位：状态首页，只看 bot 是否活着，不在这里做正式配置。

### 页面骨架

```text
+------------------------------------------------------+
| 顶部标题                                             |
+------------------------------------------------------+
| 网关状态卡片           | 系统状态卡片               |
|                        |                            |
|                        |                            |
+------------------------------------------------------+
```

### 页面模块

- `网关状态` `A/B`
  - 看网关连接状态、监听地址、接入端状态
  - 这是首页左半部分主卡片

- `系统状态` `A/B`
  - 看系统是否已配置完成、当前是否可正常运行
  - 未来再补运行时长、CPU、内存、目录占用

### 交互

- 首页只读
- 不放日志流
- 不放快捷操作
- 不放配置表单

---

## Session

定位：只管 `Session` 配置对象，不看活跃实例，不看历史对话，不放记忆系统本身配置。

### 页面骨架

```text
+----------------------+-------------------------------------------+
| Session 列表         | Session 详情                              |
|                      |                                           |
| - session A          | [基础设置]                                |
| - session B          | [消息响应]                                |
| - session C          | [上下文]                                  |
|                      | [工具]                                    |
+----------------------+-------------------------------------------+
```

### 模块 1：基础设置 `B/C`

作用：放这个 Session 最基本的工作方式。

内容：

- Session ID `B`
- 标题 / 备注 `B`
- 模板类型 `B`
- 前台人格 `B`
- 主动回复 `B`
- 唤醒限流 `C`

页面形式：

- 一组基础配置卡片
- 不单独做 tab
- `人格` 只是其中一个设置项

### 模块 2：消息响应 `B/C`

作用：定义不同事件类型到来时，这个 Session 怎么处理。

页面形式：

- 一个事件响应列表
- 每行一个事件类型
- 每行只配一个核心项：`响应方式`

响应方式：

- `respond`
- `record_only`
- `silent_drop`

群模板事件：

- 普通消息 `B`
- 被艾特 `B`
- 引用回复 `B`
- 戳一戳 `B`
- 消息撤回 `B`
- 成员加入 `B`
- 成员离开 `B`
- 管理员变更 `B`
- 文件上传 `B`
- 禁言变更 `B`
- 群荣誉变更 `B`
- 头衔变更 `B`
- 运气王 `B`
- 贴表情互动 `C`

私聊模板事件：

- 普通消息 `B`
- 引用回复 `B`
- 戳一戳 `B`
- 消息撤回 `B`
- 贴表情互动 `C`

附加项：

- 回复内容分段 `C`

### 模块 3：上下文 `C`

作用：定义这个 Session 的上下文怎么保留、怎么整理。

内容：

- 最近对话保留轮数 `C`
- 上下文整理方式 `C`

页面形式：

- 两个设置项的小卡片区
- `上下文整理方式` 对应 `截断 / 压缩`

### 模块 4：工具 `C`

作用：定义这个 Session 能调用什么，以及调用限制。

内容：

- 允许的工具 `C`
- 允许的技能 `C`
- 允许的 SubAgent `C`
- 工具调用轮数上限 `C`
- 工具调用超时时间 `C`

### 重要预留项

- Session 级对话模型链 `C`
- Session 级图片转述模型 `C`
- Session 级上下文压缩总结模型 `C`

## 记忆

定位：只管记忆系统本身，不管 Session 响应行为，不管对话历史。

### 页面骨架

```text
+------------------------------------------------------+
| Sticky Notes                                         |
| [实体列表] [当前 Note]                               |
+------------------------------------------------------+
| 长期记忆                                             |
| [基础设置] [模型] [提示词] [条目数据(后续)]          |
+------------------------------------------------------+
```

### 模块 1：Sticky Notes `A`

作用：管理用户级、会话级的便签记忆。

页面形式：

```text
+----------------------+-------------------------------------------+
| 实体列表             | 当前 Note                                 |
|                      |                                           |
| - user:...           | 只读区                                    |
| - conversation:...   | 可编辑区                                  |
|                      | 更新时间                                  |
+----------------------+-------------------------------------------+
```

内容：

- 实体分类
- 实体标识浏览 / 搜索
- 新建 Sticky Note
- 只读区
- 可编辑区
- 更新时间

### 模块 2：长期记忆设置 `A`

作用：控制长期记忆的工作方式。

内容：

- 启用长期记忆
- 提取窗口大小
- 窗口重叠大小
- 上下文注入上限

页面形式：

- 一个简洁设置区
- 不暴露存储目录这类系统级细节

### 模块 3：长期记忆模型 `B`

作用：配置长期记忆工作链路需要的模型。

内容：

- 提取模型
- 检索模型
- 向量模型

页面形式：

- 三张并列的小卡片
- 每张卡片只做绑定编辑
- 不额外展示模型内部 ID

### 模块 4：长期记忆提示词 `C`

作用：配置长期记忆专用提示词。

内容：

- 提取提示词
- 检索提示词

页面形式：

- 普通文本编辑器
- 不和通用 `提示词` 页混在一起

### 模块 5：长期记忆条目 `C`

作用：查看、筛选、编辑已经写入的长期记忆。

页面骨架：

```text
+----------------------+-------------------------------------------+
| 条目列表 / 筛选      | 条目详情                                  |
|                      |                                           |
| 搜索 / 会话 / 时间   | 主题                                      |
| 人物 / 实体 / 版本   | 事实重述                                  |
|                      | 关键词 / 人物 / 实体 / 时间 / 地点        |
| - entry A            | 依据事实                                  |
| - entry B            |                                           |
+----------------------+-------------------------------------------+
```

内容：

- 条目列表
- 关键词搜索
- 按会话筛选
- 按时间范围筛选
- 按人物筛选
- 按实体筛选
- 按提取版本筛选
- 条目详情
- 编辑主题
- 编辑事实重述
- 编辑关键词 / 人物 / 实体 / 时间 / 地点

说明：

- 这里展示的是长期记忆条目的正式字段
- 不是把底层数据库实现原样展示给用户

## 系统

定位：只放少量系统级共享配置。状态信息以首页为主，内部实现细节默认不展示。

### 页面骨架

```text
+------------------------------------------------------+
| 接入配置                                             |
| [网关配置表单]                                       |
+------------------------------------------------------+
| Catalog 扫描路径                                     |
| [Skill / SubAgent 扫描根配置]                        |
+------------------------------------------------------+
| 管理员                                               |
+------------------------------------------------------+
| 维护动作（收起）                                     |
+------------------------------------------------------+
```

### 模块 1：接入配置 `A`

作用：配置网关监听参数。

内容：

- 监听地址
- 监听端口
- 连接令牌
- 网关超时

页面形式：

- 普通配置表单
- 直接保存到网关配置
- `网关超时` 收在高级项里，不放主表单正中

### 模块 2：Catalog 扫描路径 `A/B`

作用：配置系统会去哪些目录里扫描 Skill 和 SubAgent。

内容：

- Skill 扫描路径列表
- SubAgent 扫描路径列表
- 当前是否启用 filesystem
- base_dir 说明
- 解析后的实际扫描目录预览

页面形式：

- 两组可编辑路径列表
- 每行一个路径
- 下方展示“当前实际会扫描哪些目录”的只读预览
- 保存后直接走热刷新，不要求用户额外点“重新读取配置”

说明：

- 这里改的是 catalog 扫描根，不是某个单独对象的详情
- 没显式配置时，系统回退到默认扫描目录
- 这一块属于系统级共享配置，不放进 Skill / SubAgent 页里改

### 模块 3：管理员 `A`

作用：维护共享管理员列表。

内容：

- 管理员标识列表

页面形式：

- 多行文本框
- 每行一个管理员标识

### 模块 4：维护动作 `A`

作用：执行少量维护动作，不和普通配置混在一起。

内容：

- 重新读取磁盘配置

页面形式：

- 收在单独维护区
- 配一个明确的“重新读取配置”按钮

说明：

- 这个动作只在手动改过配置文件后有意义
- 日常通过 WebUI 保存配置时，通常不需要额外点它

### 当前不放进系统页正文

- 接入是否正常
  - 这是状态，不是配置，放首页
- 当前机器人账号
  - 这是状态，不是配置，放首页
- 接入方式
  - 这是状态，不是配置，放首页
- 配置文件路径
  - 技术细节，不放正文
- 存储模式
  - 技术细节，不放正文
- WebUI 启用状态
  - 用户已经在 WebUI 里，这个信息没有意义
- Backend 会话绑定详情
  - 开发者维护信息，当前不放
- 激活模式
  - 技术概念，当前不放
- 会话绑定文件路径
  - 技术路径，当前不放
- 长期记忆存储目录
  - 系统级高级项，但当前不开放给用户改
- profiles / prompts / sessions 目录
  - 这些虽然也是 filesystem 配置，但当前还没开放成正式控制面项

## 技能

定位：技能 catalog 浏览页，展示当前被 skill catalog 追踪到的目录与文件。

### 页面骨架

```text
+----------------------+-------------------------------------------+
| 技能目录树           | 当前目录 / 文件内容                       |
|                      |                                           |
| [安装 Skill]         | /catalog-root/...                         |
| - root A (project)   | SKILL.md                                  |
|   - skill A          | references/                               |
| - root B (user)      | scripts/                                  |
|   - skill B          | assets/                                   |
+----------------------+-------------------------------------------+
```

### 模块 1：技能目录树 `A`

作用：展示当前被 skill catalog 追踪到的目录结构。

内容：

- 当前扫描根列表
- 每个扫描根的 scope
- 各个 skill 目录
- skill 文件和子目录

页面形式：

- 左侧文件树
- 类似 Finder / macOS 文件列表
- 能展开目录，不需要额外“详情页概念”

说明：

- skill 名本质上来自目录相对路径
- 页面里不强调额外展示名
- 页面主要展示目录和文件
- `project / user` 是 scope 语义，不代表页面里永远只有两棵固定目录树
- 扫描根本身在 `系统` 页配置，不在这里编辑

### 模块 2：文件内容 `A/B`

作用：查看当前选中的 skill 文件内容。

内容：

- `SKILL.md`
- `references/`
- `scripts/`
- `assets/`

页面形式：

- 右侧文件预览区
- `SKILL.md` 直接预览文本
- 其他文件先做只读浏览
- 如果需要编辑，后面再加

### 模块 3：安装 Skill `C`

作用：把 skill 安装到一个明确的默认安装根。

内容：

- 上传 zip 包
- 安装到默认安装目录

页面形式：

- 页面左上角一个 `安装 Skill` 按钮
- 选择 zip 后上传
- 安装成功后刷新目录树

说明：

- AstrBot 已经有这一套思路，支持 zip 上传后解压到默认 skills 目录
- AcaBot 当前还没有正式安装接口，所以这里是 `C`
- 第一版不做市场，不做在线安装，只做“本地 zip 安装到一个默认安装根”
- 这个“安装根”不等于任意扫描根目录，也不应让用户在安装弹窗里随便挑

## 插件

定位：运行时插件开关页。先只解决“有哪些插件、是否启用、能不能重载”。

### 页面骨架

```text
| 插件列表                                              |
| [开关] Ops Control            [保存] [重载]           |
|       插件名                                          |
|       导入路径（小字）                                |
| [开关] Backend Bridge Tool    [保存] [重载]           |
+------------------------------------------------------+
```

### 模块 1：插件列表 `A`

作用：展示当前配置里声明的插件。

内容：

- 插件显示名
- 插件名
- 导入路径（小字）
- 启用开关

页面形式：

- 单列列表
- 每个插件一行一个开关
- 每个插件单独有保存和重载动作
- 不做复杂详情页

### 模块 2：加载状态 `A`

作用：提示这条插件配置当前能不能正常加载。

内容：

- 是否可加载
- 加载失败原因

页面形式：

- 放在插件行内的小状态提示
- 出错时直接显示错误摘要

### 模块 3：维护动作 `A`

作用：对单个插件保存配置并触发重载。

内容：

- 保存当前插件配置
- 重载当前插件

说明：

- 顶部不做全局操作条
- 以单插件操作为主
- 第一版先不做插件市场
- 第一版先不做插件详情配置协议

补充说明：

- `导入路径` 是插件的技术标识
- 例如 `module.submodule:Symbol`
- 用来告诉 runtime 该从哪里加载这个插件
- 这不是主要产品信息，所以只放小字

### 当前不放进插件页

- 插件市场
- 在线安装
- 版本管理
- 细粒度权限
- 独立插件详情页

## 工具

定位：工具目录页，只看当前已经注册了哪些工具，不在这里配置人格可见性。

### 页面骨架

```text
+----------------------+-------------------------------------------+
| 工具列表             | 工具详情                                  |
|                      |                                           |
| - read               | 名称                                      |
| - write              | 来源                                      |
| - bash               | 说明                                      |
| - ask_backend        | 参数定义                                  |
+----------------------+-------------------------------------------+
```

### 模块 1：工具列表 `A`

作用：展示当前已经注册的工具目录。

内容：

- 工具名
- 工具来源

页面形式：

- 左侧列表
- 按来源分组
  - 内置工具
  - 外来工具
- 不提供开关

### 模块 2：工具详情 `A`

作用：查看一个工具的大致能力。

内容：

- 名称
- 来源
- 说明
- 参数定义

页面形式：

- 右侧详情区
- 参数定义用只读结构化表单展示

说明：

- 当前工具目录能稳定拿到的是 名称、说明、来源、参数定义 和少量附加信息
- 插件注册的工具会直接出现在这里
- 这页只做目录查看，不做编辑

### 当前不放进工具页

- 给人格启用工具
- 给 Session 启用工具
- 工具权限策略
- 工具安装

说明：

- 这些都不是“工具目录”本身的职责
- 启用关系属于人格或 Session 配置

## SubAgent

定位：子代理目录。它和 `Skill` 很像，都会在系统提示词里声明“有哪些可以用”，但调用方式不同：

- `Skill` 用 `Skill` 工具调用
- `SubAgent` 用 `delegate_subagent` 工具调用

`SubAgent` 不是插件对象，也不是工具对象。
当前页面先按“catalog 里的子代理清单”来理解，不等同于未来更强的可编辑对象管理页。

### 页面骨架

```text
+----------------------+-------------------------------------------+
| SubAgent 列表        | SubAgent 详情                             |
|                      |                                           |
| - excel-worker       | 名字 / 描述                              |
| - browser-helper     | 工具 / 来源 / 生效状态                   |
| - note-organizer     | 路径 / model_target                      |
+----------------------+-------------------------------------------+
```

### 模块 1：子代理列表 `A/B`

作用：先给主 agent 和用户看“当前有哪些可委派子代理”。

内容：

- subagent_name
- 可选的简短描述
- 生效状态摘要

页面形式：

- 左侧列表
- 默认突出名字
- 可以配很轻的辅助信息，但不需要堆满实现细节

说明：

- 列表页只负责选中对象
- 详细信息放右侧详情区

### 模块 2：子代理详情 `A/B`

作用：展开看这个子代理是谁。

内容：

- subagent_id
- subagent_name
- description
- tools
- source
- host_subagent_file_path
- effective
- model_target

页面形式：

- 右侧详情区
- 顶部只看名字和描述
- 下方展示工具列表、来源、路径和生效状态

说明：

- 这里展示的是当前 catalog manifest 已经稳定提供的信息
- 不是去反推插件注册表，也不是去反推底层执行实现元数据
- `source / path / effective` 属于当前用户理解“为什么有两条同名 SubAgent”所必需的信息

### 当前不放进 SubAgent 页

- 新建子代理
- 删除子代理
- 开关子代理
- 委派策略配置
- 完整权限配置
- SUBAGENT.md 正文编辑
- 直接改扫描路径

说明：

- `执行器` 是实现层术语，不在页面上出现
- 当前这页是“catalog 清单查看页”，不是完整对象管理页
- 真正还没完成的是“可编辑的正式 SubAgent 对象层”

### 当前已完成的代码改造

- `SubAgent` 已经来自正式 catalog，不再来自插件注册表
- `delegate_subagent` 已经面向 catalog 解析
- system prompt 里的可见子代理摘要已经来自 catalog
- 插件侧已经移除“直接提供 SubAgent”这条产品语义

### 后续代码改造方向

目标边界：

- `插件` 仍然是通用扩展点
- `SubAgent` 是正式对象
- `SubAgent` 可以分配工具和 Skill
- 主 agent 看到的子代理列表来自正式子代理对象，不来自插件注册表

改造步骤：

1. 新增真正可编辑的 `SubAgent` 对象层
   - 至少要有：
     - `委派 ID`
     - `name`
     - `description`
     - `enabled_tools`
     - `skills`

2. 子代理自己的工具和 Skill 分配独立管理
   - 主 agent 和 SubAgent 都从统一工具层取能力
   - 但分配关系分别配置，不互相混用

3. 页面从“catalog 查看”升级到“对象编辑”
   - 当前详情页展示 manifest 信息
   - 后续再补真正的保存、校验、删除与创建流程
