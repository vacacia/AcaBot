# Phase 10: Group Chat Bug Fix - Research

**Researched:** 2026-04-05
**Domain:** AcaBot session surface config naming mismatch
**Confidence:** HIGH

## Summary

Group chat bot replies to ALL messages instead of filtering by admission config. Root cause is a naming mismatch: the three group session YAML files use surface names (`message`, `message_mention`, `message_reply`) that do not match the surface candidate chain generated by `_surface_candidates()` (`message.plain`, `message.mention`, `message.reply_to_bot`). `resolve_surface()` finds no match, falls back to a non-existent `message.plain`, and `_surface_config()` returns an empty `SurfaceConfig()` ŌĆö causing `resolve_admission()` to default to `mode: respond`.

Fix is to rename three surface keys in each of the three group session YAML files. No code changes needed.

## Common Pitfalls

### Pitfall 1: YAML Key Rename Only (Partial Fix)
**What goes wrong:** Only `message` renamed to `message.plain`, but `message_mention` and `message_reply` left unchanged.
**How to avoid:** Rename ALL THREE surface keys in each of the three files. A test run with `@` mention will still fall through to the default `respond`.

### Pitfall 2: Wrong YAML Indentation
**What goes wrong:** YAML is whitespace-sensitive. `message.plain:` must be at the same level as `message:` was.
**How to avoid:** The surfaces block structure stays identical ŌĆö only the key name changes. Verify the `surfaces:` parent key aligns with its children.

### Pitfall 3: Only Fixing Test Group
**What goes wrong:** Only `1097619430` fixed, leaving `1039173249` and `742824007` still buggy.
**How to avoid:** All three groups were confirmed to have the wrong surface naming. All three must be fixed.
