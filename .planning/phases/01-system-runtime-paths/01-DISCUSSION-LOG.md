## 路径信息展示

| Option | Description | Selected |
|--------|-------------|----------|
| 正文直接展示 | 把路径和技术细节直接放到系统页正文主区域 | |
| 高级区展示，技术信息翻译成人话 | 技术细节可以展示，但放到高级区，并用操作者能懂的标签表达 | ✓ |
| 尽量隐藏技术细节 | 路径等技术信息尽量不在系统页露出 | |

**User's choice:** Phase 1 按建议方案做。

## 保存与生效语义

| Option | Description | Selected |
|--------|-------------|----------|
| 保存即尝试生效 | 正常保存后立即尝试应用；只有特殊项才提示重启 | ✓ |
| 保存和 apply 分离 | 先写配置，再手动点 apply/reload | |
| 只写回文件 | 不承诺立刻应用，全靠维护动作同步 | |

**User's choice:** 尽量让全部内容都保存即尝试生效。  
**Notes:** gateway 这类项允许“已保存，但需重启后生效”；维护动作区仍保留，但不成为普通保存流程的一部分。

## 列表输入交互

| Option | Description | Selected |
|--------|-------------|----------|
| 多行 textarea | 一行一个值，保存时再统一 split/trim/filter | |
| 单项列表编辑 | 一次填一个，单项校验、逐项删除，必要时可补批量导入 | ✓ |
| 自由结构化文本 | 允许用户直接输入结构化块 | |

**User's choice:** 列表项改成一次填一个的形式，参考讨论里给的示例图。  
**Notes:** 用户明确希望把“去空行、去重”这类事更多前移到交互设计里，避免依赖 multiline 文本输入后的集中清洗。

## 校验与错误反馈

| Option | Description | Selected |
|--------|-------------|----------|
| 强前端阻断 | 前端先判断路径存在/可读，不通过就不让提交 | |
| 前端 warning 但允许保存 | 前端先做更重的路径检测，展示 warning，再允许继续 | |
| 后端权威校验 | 前端只做轻量输入校验；路径存在/可读由后端在真实上下文里判断 | ✓ |

**User's choice:** 如果前端做这类路径校验会明显变麻烦，就不要强做。  
**Notes:** 结合当前架构，最终收束为“前端只做轻量单项输入校验；路径存在/可读以后端和 control plane 的真实解析结果为准；错误用人话主结论 + 技术细节展开区呈现”。
